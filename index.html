You're going to build a working app. Locally. Today.

This guide walks you through the whole thing in 8 simple steps. No prior coding experience needed — you'll install a few tools, paste one prompt into Claude Code, run the app on your computer, test it, and commit your work. Someone else will handle deployment.

⏱ About 1 hour 🧰 8 steps 🤖 1 prompt to Claude Code 💻 Works on Windows

After everything is installed

Open your Start Menu, type cmd, and press Enter to open a Command Prompt. Paste these two commands, pressing Enter after each. You're telling your computer to trust the development certificate your app will use.

dotnet dev-certs https --trust
git --version

If the first command asks for permission, click Yes. The second should print something like git version 2.45.x. If it does, you're ready.

Done with Step 1. You only do this once. From now on your computer is set up for building.
2

Get the project onto your computer

Two quick parts

This step has two halves. First you'll make your own copy of the project skeleton on GitHub (a couple of clicks — no code). Then you'll clone it to your computer so you can open it in Visual Studio.

Part A — Create YOUR repo from the template (in your browser)

Your integrator has prepared a starter project that already contains the correct folder structure, packages, and settings. You don't have to build this from scratch — you just click a button and GitHub makes a fresh copy for you.

Open the starter template in your browser

Go to the URL your integrator gave you. It looks like:

https://github.com/lfz-ai-powered-app/portal-template

If GitHub asks you to sign in, do that first.

Click the green "Use this template" button

It's near the top of the page, next to the blue Code button. Click it, then click Create a new repository in the dropdown.

Fill in the short form

Owner: lfz-ai-powered-app (select it from the dropdown).

Repository name: a short lowercase name like my-portal-app or warehouse-locations. Ask your integrator what to call it if unsure.

Private: leave this selected.

Click "Create repository"

GitHub creates your new repo in a few seconds. The page will refresh and show it — it's now yours, with all the starter code inside.

GitHub — portal-template repo
The portal-template repo on GitHub. The green Use this template button creates a fresh copy for you in seconds.
Why this works so well: your new repo has all five projects, all settings, and a clean commit history — as if you'd built it yourself. No "template" label lingers; it's genuinely yours.

Part B — Clone YOUR repo to your computer (in Visual Studio)

Your new repo lives on GitHub. Now you bring a copy onto your machine so you can edit and run it.

Copy the URL of the repo you just created — it looks like:

https://github.com/lfz-ai-powered-app/my-portal-app

"Cloning" means downloading a copy that stays linked back to GitHub so your changes can sync.

Open Visual Studio

Start Menu → type Visual Studio 2022 → Enter.

On the welcome window, click "Clone a repository"

It's one of four big buttons on the right. If you missed the welcome screen, use the menu: File → Clone Repository…

Paste YOUR new repo URL

Paste the URL (the one for the repo YOU just created, not the template) into the top box. Visual Studio auto-fills the local folder — something like C:\Users\You\source\repos\my-portal-app. Leave that as-is.

Click "Clone"

If GitHub asks you to sign in, click Sign in with browser, approve the popup, then come back to Visual Studio.

Visual Studio 2022 — Start Window
On the Visual Studio start window, click Clone a repository (top option on the right).
The project should now open automatically. If you see a tree of files on the right (the Solution Explorer), you're in good shape.

4

Tell Claude Code what to build

The big step

This is where the magic happens. Open Claude Code inside your project folder (the same folder Visual Studio is showing). Then paste the prompt below.

Open a Command Prompt in your project folder

In Visual Studio: Tools → Command Line → Developer Command Prompt. That terminal is already pointing at your project folder.

Or: File Explorer → navigate to your project folder → click the address bar → type cmd → press Enter.

Start Claude Code

In that terminal, type claude and press Enter. Claude Code opens in the same window.

Edit one paragraph of the prompt below

Replace the text in the FEATURE DESCRIPTION section with what your app needs to do. Everything else stays exactly as it is — it locks in the correct technology stack so your code can be integrated later.

Copy the prompt → paste into Claude Code → press Enter

Use the big orange Copy button on the card below. Claude will ask you clarifying questions, then start generating files. Answer plainly — e.g. "yes, include a date column" or "no, skip the delete button".

5

Run the app on your computer

Press Play

Switch back to Visual Studio. Your project now has a bunch of new files — that's normal.

Set the startup projects

In Solution Explorer, right-click the solution (the very top row, usually bold). Click Configure Startup Projects… → select Multiple startup projects → find the two ending in .API and .Web → set both to Start → click OK.

Press Ctrl + F5

This runs the app without debugging — it starts much faster than regular F5. Two browser tabs should open after a few seconds.

You should see two things

Tab 1: a page titled "Swagger" at https://localhost:7243 — that's your back-end. You can ignore it.

Tab 2: your actual app at https://localhost:7154 — that's what you'll test.

Solution Property Pages — Configure Startup Projects
Pick Multiple startup projects, then change MyPortal.API and MyPortal.Web to Start. Click OK.
If the browser shows "Your connection is not private", click Advanced → Proceed to localhost. That's because your dev certificate isn't trusted by the browser yet — safe on your own machine.

6

Test what you built

Sanity check

Click through your app like a real user would. Try every button. Try the thing you described to Claude. If anything looks wrong, go back to Claude Code and say so in plain English — e.g. "the Save button doesn't work" or "the date shows incorrectly". Claude will fix it.

Before you move on, make sure…

How to stop the app

Switch back to Visual Studio and press the red Stop button (or close the browser tabs). If Visual Studio itself won't stop, press Shift + F5.

7

Commit your work to GitHub

Save your progress

"Committing" means saving a snapshot of your changes to GitHub so the integrator can pick them up. Visual Studio has a friendly UI for this — you never need to touch the command line.

Option A — The easy way (Visual Studio's Git panel)

Open the Git Changes panel

Menu: View → Git Changes (or the shortcut Ctrl+0, then Ctrl+G). You'll see a list of files you changed.

Create a new branch

At the top of the panel, click the current branch name (usually main) → New Branch → name it something like feat/my-feature → click Create Branch.

Why: you never commit directly to main. Always work in a branch.

Write a commit message

In the big text box at the top of Git Changes, type a short sentence describing what you did. Examples:

• "Add page for managing warehouse locations"
• "Fix the Save button on the Edit page"
• "Apply LFZ colours to the navigation bar"

Click Commit All

That button is right below the message box. A green bar confirms it worked.

Click Push

You'll see a "Push" button in the top bar of the Git Changes panel (an up-arrow icon). Click it. Visual Studio uploads your branch to GitHub.

Open a Pull Request

After pushing, a yellow "Create a pull request" bar appears. Click it — it opens GitHub in your browser with a form. Write a sentence about what you built, then click Create pull request.

A Pull Request is just a formal "please review my work" notification to the integrator.

Visual Studio — Git Changes panel
The Git Changes panel in Visual Studio. Type a message → click Commit All → then click the up-arrow at the top to Push.

After pushing — GitHub asks you to open a Pull Request

GitHub — Open a pull request
On GitHub, write a short title + a few lines describing what you built. Then click the green Create pull request button. Done.

Option B — If you prefer the terminal

Ask Claude Code to do it for you. Paste this into Claude Code:

Please do the following for me:
1. Show me what I've changed (git status).
2. Create a new branch called feat/<short-name> (ask me for a name if you're not sure).
3. Stage every file EXCEPT secrets, .env, *.pfx, bin/, obj/.
4. Commit with a clear one-line message describing what we built.
5. Push the branch to GitHub.
6. Open a Pull Request for me using `gh pr create`.

After each step, tell me what happened in plain English.
One golden rule: never commit files containing passwords, API keys, or connection strings. If Claude asks about a file that looks sensitive, say skip it.

If something breaks

Relax

You will hit a problem at some point — everyone does. The fix is nearly always one of these:

Copy the error, paste it into Claude Code

Say "this error happened when I did X". Claude fixes most things in one or two rounds.

Stop and re-run the app

Press the red Stop button in Visual Studio, then press Ctrl+F5 again. Fixes 30% of weird issues.

Close and reopen Visual Studio

If it's acting weird or showing stale files, restart it. Nothing is lost.

Ask your integrator

Stuck for more than 30 minutes? Message them. They've seen most of the problems before.

Common issues, explained

Browser says "Your connection is not private"

Click Advanced → Proceed to localhost. Or re-run dotnet dev-certs https --trust in Command Prompt.

The app takes 15–20 seconds to start the first time

Normal for the very first run — your computer is compiling things. Subsequent runs are fast. Always use Ctrl+F5, never F5 alone (debug mode is 3–4× slower).

"Cannot connect to SQL Server"

SQL LocalDB sometimes goes to sleep. Open Command Prompt and run: sqllocaldb start MSSQLLocalDB. Then press Ctrl+F5 again.

"Port 7243 is already in use"

Another instance is still running. Open Task Manager (Ctrl+Shift+Esc), find anything with "dotnet" or your project name, end those tasks, and re-run.

Git Changes is empty after Claude worked

In Visual Studio: View → Git Changes. If still empty, close and reopen Visual Studio — it'll pick up the changes on disk.

Glossary

Jargon-free

Terms you'll see people throw around. No need to memorise — skim and come back when you need it.

TermWhat it means in plain English

PushUpload your commits to GitHub so others can see them.
